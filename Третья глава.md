# Глава 3. Технологическая реализация и экспериментальная оценка системы

В третьей главе описывается практическая реализация прототипа и экспериментальная проверка качества детекции. Текст опирается на структуру проекта и результаты тестового прогона, выполненного на синтетических сценариях. Цель главы — показать, как проектная архитектура из второй главы реализована в коде, какие решения приняты на каждом этапе, и какие метрики получены при проверке качества.

## 3.1 Выбор технологического стека

Технологический стек подбирался исходя из двух ключевых требований: воспроизводимость эксперимента и полнота конвейера от захвата пакетов до расчёта метрик качества. Именно поэтому выбор пал на Python как базовый язык: он предоставляет зрелые библиотеки для захвата трафика, обработки данных и машинного обучения в едином экосистемном пространстве.

Таблица 3.1 — Технологический стек прототипа

| Уровень системы | Выбранный инструмент | Обоснование выбора |
|---|---|---|
| Захват трафика | Scapy + Npcap (Windows) | Гибкость разбора пакетов, поддержка BPF-фильтров, не требует внешних зависимостей для формата данных |
| Хранение данных | PyArrow / Parquet | Колонночный формат, партиционирование по `date/hour`, эффективное чтение подмножества данных |
| Обработка данных | Pandas, NumPy | Стандартный стек для агрегации, нормализации и вычисления статистических признаков |
| ML-модели | PyOD (IForest, AutoEncoder) | Единый API для нескольких алгоритмов без учителя, удобное сравнение моделей |
| Масштабирование | scikit-learn StandardScaler | Артефакт сохраняется вместе с моделью, гарантирует воспроизводимость нормализации |
| Конфигурация | PyYAML | Все управляемые параметры вынесены в `pipeline_config.yaml` |
| Мониторинг | prometheus-client | Минимальная зависимость, экспорт метрик в стандартном формате для Prometheus/Grafana |
| Оркестрация | `main_pipeline.py` | Единая точка запуска всех этапов; поддержка отдельного запуска каждого |

![Технологический стек прототипа системы детекции](figures/chapter3/figure_3_1_tech_stack.png)

Рисунок 3.1 — Технологический стек прототипа системы детекции.

Отдельно обоснован выбор безучительской постановки. Разметка атак в реальных сетях дорога и редко полна. PyOD позволяет обучиться только на нормальном трафике и затем ранжировать отклонения через `decision_function`. Это делает систему применимой с первого дня эксплуатации без требования к предварительно собранному корпусу инцидентов.

Ключевое инженерное решение — разделение захвата и анализа на два независимых процесса. Коллектор пишет батчами в Parquet и не блокируется тяжёлой обработкой. Анализ читает готовый датасет и может быть повторён без повторного захвата трафика. Это минимизирует потери при высокой нагрузке и делает эксперименты дешёвыми: изменение набора признаков не требует нового сеанса захвата.


## 3.2 Реализация коллекторов и транспортного слоя

Коллектор реализован в модуле `Packet-Real-Time-Collector/src/main.py` и состоит из двух взаимодействующих классов: `PacketCapture` и `DatasetWriter`.

`PacketCapture` вызывает `scapy.sniff()` с опциональным BPF-фильтром (`tcp or udp` по умолчанию) и для каждого перехваченного пакета извлекает метаданные. Полный payload не сохраняется: вместо него в поле `payload_sample` записывается не более 32 байт в hex-представлении. Это позволяет рассчитать энтропию короткого образца при углубленной проверке, не нарушая требований приватности.

Для каждого пакета формируется запись из 19 полей:

Таблица 3.2 — Поля записи коллектора

| Поле | Тип | Описание |
|---|---|---|
| `ts_us` | int64 | Метка времени в микросекундах |
| `date`, `hour` | string | Партиционные ключи (UTC) |
| `iface` | string | Сетевой интерфейс |
| `length`, `payload_len` | int32 | Длина пакета и полезной нагрузки |
| `direction` | string | `in` / `out` по MAC интерфейса |
| `eth_type` | string | Тип Ethernet-кадра в hex |
| `src_mac`, `dst_mac` | string | MAC-адреса |
| `ip_version` | int32 | 4 или 6 |
| `protocol` | string | TCP, UDP или числовой код |
| `src_ip`, `dst_ip` | string | IP-адреса источника и назначения |
| `src_port`, `dst_port` | int32 | Порты транспортного уровня |
| `tcp_flags` | string | Флаги TCP (например, `S`, `SA`) |
| `flow_key` | string | `src_ip:sport-dst_ip:dport-protocol` |
| `payload_sample` | string | Первые 32 байта payload в hex |

`DatasetWriter` работает в отдельном daemon-потоке. Он принимает записи через очередь с лимитом 20 000 элементов и скидывает их в Parquet батчами по `batch_size=1000` записей или принудительно каждые `flush_interval=2.0` секунды. При переполнении очереди новые записи тихо отбрасываются: захват трафика никогда не блокируется записью. Для промышленного использования этот случай необходимо сопровождать метрикой потерь; в прототипе он считается приемлемым риском на высоких нагрузках.

Партиционирование `date=YYYY-MM-DD/hour=HH` даёт естественную гранулярность для экспериментов: любой час доступен как самостоятельная единица анализа без перечитывания всего архива. Это особенно важно при сравнении нескольких временных интервалов в одном эксперименте.

Для тестирования без реального трафика реализован модуль `synthetic_generator.py`. Он производит два типа синтетических данных:
- **Нормальный трафик** — случайные пары из пулов `192.168.1.10–50` (источники) и `10.0.0.100–150` (назначения), порты из списка стандартных сервисов, метка `is_anomaly=False`.
- **Burst-аномалия** — к нормальному трафику добавляется x5 пакетов с увеличенным размером в заданный временной интервал, `is_anomaly=True`.
- **Port scan** — 20 TCP SYN-пакетов в секунду на случайные порты 1–1024 одного целевого узла, `is_anomaly=True`.

Синтетический генератор записывает результат в тот же формат Parquet с теми же партициями, что и реальный коллектор. Это позволяет использовать один и тот же pipeline предобработки для обоих источников.

## 3.3 Реализация конвейера предобработки и feature engineering

Предобработка реализована в модуле `detection_core.py` и вызывается из `main_pipeline.py` на этапе `preprocess`. Результат каждого прогона сохраняется в `data/processed/frames_<timestamp>.pkl` и в `data/processed/features.csv`.

**Шаг 1 — Чтение данных.** Pipeline автоматически находит последний по времени изменения Parquet-файл в `data/raw/` или `data/synthetic/`. Это делает этап самодостаточным: при добавлении новых данных не нужно менять конфигурацию.

**Шаг 2 — Группировка по потокам.** Функция `flow_rows_from_parquet(df, window_size)` группирует строки по полю `flow_key`. Внутри каждой группы пакеты сортируются по `ts_us`. Далее каждый поток нарезается на фреймы по `window_size=100` наблюдений. Параметр задаётся в `pipeline_config.yaml` и применяется одинаково ко всем экспериментам.

**Шаг 3 — Извлечение признаков.** Функция `extract_frame_features` вычисляет 15 числовых признаков и категориальное поле `protocol` для каждого фрейма:

Таблица 3.3 — Признаки фрейма, вычисляемые при предобработке

| Группа | Признаки | Связанный сценарий |
|---|---|---|
| Объём потока | `packet_count`, `total_bytes` | Burst, флуд, массовые передачи |
| Временны́е | `duration`, `avg_inter_arrival`, `std_inter_arrival` | Ритм обмена, автоматизированные серии |
| Размеры пакетов | `avg_packet_size`, `std_packet_size`, `min_packet_size`, `max_packet_size` | Нестандартные сессии, туннелирование |
| Интенсивность | `pps`, `bps` | Всплески нагрузки, DoS-подобные режимы |
| Транспортный контекст | `src_port`, `dst_port`, `syn_flag_ratio` | Port scan, SYN-серии |
| Payload | `avg_payload_entropy` | Косвенная оценка шифрования/сжатия |

Для вычисления `pps` и `bps` длительность ограничена снизу значением `EPS_DUR = 0.001` секунды, чтобы исключить деление на ноль при одиночных пакетах. Это решение зафиксировано явно: одиночные пакеты получают завышенные значения интенсивности, что учитывается при интерпретации score.

Метка `is_anomaly` переносится из исходных данных: фрейм считается аномальным, если хотя бы один пакет в нём имеет `is_anomaly=True`. Это правило корректно для burst-сценария и достаточно для базовой оценки качества.

**Шаг 4 — Нормализация.** `StandardScaler` обучается на тренировочном наборе нормальных фреймов и сохраняется в `models/scaler.joblib` как отдельный артефакт. При инференсе используется тот же scaler, что гарантирует единообразие нормализации между обучением и применением.

**Шаг 5 — Удаление выбросов.** При включённом `outlier_removal: true` к тренировочному набору нормы применяется IQR-фильтрация (1.5×IQR) по каждому числовому признаку. Это снижает влияние редких законных всплесков (например, резервного копирования) на форму базовой линии.

## 3.4 Реализация и настройка ML-моделей

ML-модели реализованы в `detection_core.py`. Центральным объектом является датакласс `TrainedEnsemble`, который хранит всё состояние, необходимое для инференса: scaler, модели, пороги и метаданные версии.

```python
@dataclass
class TrainedEnsemble:
    scaler: StandardScaler
    iforest: Optional[Any]       # IForest из PyOD
    autoencoder: Optional[Any]   # AutoEncoder из PyOD (опционально)
    feature_columns: List[str]
    contamination: float         # 0.1
    threshold_quantile: float    # 0.99
    t_low: float                 # 90-й квантиль score на обучающих данных
    t_high: float                # 99-й квантиль score на обучающих данных
    model_version: str           # "pyod_iforest+autoencoder.v1"
```

**Режим `hybrid` (по умолчанию)** обучает оба компонента: `IForest(n_estimators=200, contamination=0.1)` и `AutoEncoder(epochs=50, contamination=0.1)`. Если AutoEncoder недоступен (ошибка импорта или обучения), система gracefully деградирует до одного IForest без потери работоспособности.

**Режим `autoencoder`** пробует сначала обучить AutoEncoder, а при неудаче — IForest как резерв.

Итоговый `anomaly_score` вычисляется как среднее min-max нормированных `decision_function` обеих моделей:

```
score_i = mean(minmax(iforest.decision_function), minmax(ae.decision_function))
```

Нормировка в [0, 1] делает score интерпретируемым независимо от масштаба каждой модели.

**Пороги.** После обучения на нормальных фреймах вычисляются:
- `t_low` = 90-й квантиль score обучающей выборки
- `t_high` = 99-й квантиль (управляется `threshold_quantile` в конфигурации)

Такой способ привязывает пороги к реальному распределению нормального трафика, а не к абстрактному числу. В тестовом прогоне получено: `t_low = 0.611`, `t_high = 0.858`.

**Сохранение и загрузка.** Функции `save_ensemble` и `load_ensemble` сериализуют весь объект `TrainedEnsemble` через `pickle` в `models/ensemble.joblib`. Это единый файл, содержащий и scaler, и модели, и пороги. При инференсе достаточно загрузить один артефакт.

## 3.5 Реализация слоя принятия решений и политик реакции

Слой принятия решений реализован двумя функциями в `detection_core.py`: `deep_inspection` и `decide_status`.

**`deep_inspection(row, score)`** проверяет каждый фрейм по набору детерминированных правил и возвращает список строковых свидетельств (`evidence`):

Таблица 3.5 — Правила углубленной проверки

| Правило | Условие срабатывания | Что сигнализирует |
|---|---|---|
| `rule:tcp_syn_low_ports_short_flow` | TCP + dst_port < 1024 + syn_flag_ratio ≥ 0.5 + packet_count ≤ 5 | Короткая SYN-попытка на привилегированный порт |
| `rule:high_intensity_burst` | pps ≥ 8 000 или bps ≥ 80 Мбит/с | Высокоинтенсивный всплеск |
| `rule:high_entropy_payload_sample` | avg_payload_entropy ≥ 7.2 + packet_count ≥ 3 | Вероятно зашифрованные или сжатые данные |
| `rule:compressed_timeline_many_packets` | duration ≤ 0.01 с + packet_count ≥ 20 | Аномально высокая плотность пакетов |
| `signal:high_ml_score` | score ≥ 0.85 | Сильный сигнал от ML-модели |

**`decide_status(score, t_low, t_high, evidence)`** присваивает финальный статус:

| Условие | Статус | Смысл |
|---|---|---|
| score < t_low и evidence пуст | `regular` | Нет признаков отклонения |
| score ≥ t_low или есть любое свидетельство | `suspicious` | Повышенный риск, требует внимания |
| score ≥ t_high и есть хотя бы одно `rule:*` | `malicious` | Подтверждённое высокорисковое событие |

Принципиальный момент: статус `malicious` требует одновременно высокого ML-score **и** правилового подтверждения. Это сознательная защита от ложных блокировок: только ML-сигнал недостаточен для максимального статуса.

Функция `predict_frames` объединяет всё: инференс ансамбля, углубленную проверку и присвоение статуса. Результат сохраняется как DataFrame, где каждая строка — это структурированное событие:

Таблица 3.6 — Поля события на выходе `predict_frames`

| Поле | Содержание |
|---|---|
| `ts` | Время расчёта события (UTC ISO 8601) |
| `frame_key`, `flow_key` | Идентификатор фрейма и потока |
| `segment` | Сетевой сегмент (интерфейс) |
| `status` | `regular` / `suspicious` / `malicious` |
| `anomaly_score` | Числовая оценка [0, 1] |
| `thresholds` | Использованные пороги T_low и T_high |
| `evidence` | Список сработавших правил |
| `top_features` | Топ-5 признаков по абсолютному значению |
| `context` | src_ip, dst_ip, src_port, dst_port, protocol |
| `model_version` | Версия ансамбля для аудита |

Результат записывается в `results/events_<timestamp>.jsonl`. Каждая строка — валидный JSON-объект, пригодный для прямой загрузки в SIEM или систему корреляции.

## 3.6 Реализация мониторинга и дашбордов

Мониторинг реализован в `monitoring_system.py` через `prometheus-client`. При запуске `MonitoringSystem.start()` поднимается HTTP-сервер на порту `9108`, который экспортирует метрики в формате Prometheus text.

Таблица 3.7 — Метрики, экспортируемые `MonitoringSystem`

| Метрика | Тип | Что измеряет |
|---|---|---|
| `ndr_packets_captured_total` | Counter | Суммарное число записанных пакетов |
| `ndr_frames_processed_total` | Counter | Суммарное число обработанных фреймов |
| `ndr_pipeline_latency_ms` | Histogram | Сквозная задержка обработки (бакеты 10–5000 мс) |
| `ndr_decisions_total{status}` | Counter | Число вердиктов с разбивкой по статусу |
| `ndr_anomaly_score` | Histogram | Распределение anomaly score (бакеты 0.05–1.0) |
| `ndr_suspicious_rate` | Gauge | Доля non-regular фреймов в последнем батче |

Метод `observe_batch(latency_ms, scores, statuses)` вызывается после каждого батча инференса и обновляет все метрики за один вызов. Это снижает накладные расходы и обеспечивает атомарность обновления.

Если `prometheus-client` не установлен, `MonitoringSystem` работает в режиме заглушки: все методы вызываются без побочных эффектов. Это позволяет запускать систему без зависимости от Prometheus в средах, где мониторинг не требуется.

Целевая схема визуализации предполагает Grafana-дашборд с четырьмя панелями (Рисунок 2.6): временной ряд `ndr_suspicious_rate`, гистограмма `ndr_anomaly_score`, счётчик вердиктов по статусам и гистограмма задержек `ndr_pipeline_latency_ms`. Такая конфигурация позволяет на одном экране диагностировать как деградацию качества (рост доли suspicious/malicious), так и деградацию производительности (рост latency).

Ключевой сценарий использования мониторинга — контроль дрейфа. Если `ndr_suspicious_rate` резко вырос без изменений в конфигурации, это может означать либо реальную аномальную активность, либо изменение фонового трафика, которое требует переобучения модели. Разграничить эти два случая позволяет совместный анализ `ndr_anomaly_score` (изменилось ли распределение score) и `ndr_decisions_total` (выросли ли вердикты `malicious` с правиловым подтверждением).

## 3.7 Подготовка датасета и экспериментального стенда

Для проверки качества детекции использован синтетический датасет, сгенерированный встроенным `synthetic_generator.py`. Такой подход обеспечивает воспроизводимость: момент начала аномалии, её длительность и интенсивность задаются параметрами генерации, а метка `is_anomaly` расставляется программно.

**Сценарий burst (основной).** Генерируется нормальный трафик продолжительностью 60 секунд с интенсивностью 10 пакетов/с. На интервале 30–40 секунд к нормальному трафику добавляется burst: для каждого исходного пакета создаётся ещё 4 копии с увеличенным размером (1000–1500 байт), итого ×5 от нормальной интенсивности. Аномальные записи размечены `is_anomaly=True`.

**Сценарий port scan (дополнительный).** Генерируется 20 TCP SYN-пакетов/с к одному целевому узлу на случайные привилегированные порты (1–1024) в течение 30 секунд. Этот сценарий создаёт аномалии структурного типа: не по интенсивности, а по комбинаторике попыток соединения.

Таблица 3.8 — Параметры экспериментального стенда

| Параметр | Значение |
|---|---|
| Нормальный трафик | 60 сек × 10 pps = 600 записей |
| Burst-аномалия | 30–40 сек, множитель ×5 |
| Источники в нормальном трафике | `192.168.1.10–50` → `10.0.0.100–150` |
| window_size (предобработка) | 100 |
| contamination | 0.1 |
| threshold_quantile | 0.99 |
| Разбивка train/test | Нормальные фреймы — обучение; все фреймы — валидация |

Правило разметки фрейма как аномального: фрейм считается аномальным, если хотя бы один пакет в нём имеет `is_anomaly=True`. Это консервативное правило, которое не наказывает «переходные» фреймы на границе burst-интервала: если хотя бы один аномальный пакет попал в окно, оно считается аномальным.

Полный экспериментальный стенд выполняется командой:

```bash
python main_pipeline.py --stage all --model-type hybrid
```

Последовательность: генерация → preprocessing → train → validate. Все артефакты сохраняются в `data/processed/`, `models/`, `results/`. При повторном запуске pipeline автоматически находит последние по timestamp файлы, что исключает ручное указание путей.


## 3.8 Метрики оценки качества детекции

Для оценки качества используется двухуровневая система метрик. Первый уровень — метрики ранжирования, не зависящие от конкретного порога. Второй уровень — метрики на выбранном пороге, отражающие эксплуатационную реальность.

Таблица 3.9 — Метрики качества детекции

| Метрика | Уровень | Что показывает | Почему важна |
|---|---|---|---|
| ROC-AUC | Ранжирование | Вероятность, что модель ранжирует случайную аномалию выше нормы | Оценка модели независимо от порога |
| PR-AUC | Ранжирование | Среднее precision при разных порогах recall | Устойчива к сильному дисбалансу классов |
| Recall | Порог | Доля обнаруженных аномалий | Риск пропуска инцидента |
| Precision | Порог | Доля верных срабатываний среди всех | Нагрузка на оператора |
| F1 | Порог | Гармоническое среднее precision и recall | Удобен для сравнения методов |
| FPR | Порог | Доля ложных срабатываний | Ключевой эксплуатационный риск |

Выбор PR-AUC как одной из основных метрик обоснован природой задачи: аномальные фреймы составляют значительно меньшую часть потока по сравнению с нормальными. В таких условиях высокая accuracy ничего не означает, а ROC-AUC может давать завышенную оценку. PR-AUC корректно отражает качество именно при редком классе.

FPR вынесен в эксплуатационные критерии по отдельной причине: в реальном SOC ложное срабатывание стоит дороже одного пропущенного слабого сигнала. Поток ложных алертов приводит к «усталости от оповещений» и фактической деградации процесса реагирования. Поэтому контроль FPR при фиксированном пороге рассматривается как обязательное требование.

Дополнительно фиксируются эксплуатационные метрики: задержка обработки (мс на цикл), доля аномальных событий в потоке и стабильность score между прогонами. Они позволяют обнаружить деградацию системы ещё до падения метрик качества.

**Интерпретация результатов.** Recall 0.97 означает, что модель обнаруживает практически все аномальные фреймы в burst-сценарии. Это ожидаемо: burst создаёт сильный сигнал по `pps`, `bps` и `total_bytes`, которые входят в базовый вектор признаков.

ROC-AUC 0.955 и PR-AUC 0.856 показывают, что ранжирующая способность модели высокая: аномальные фреймы статистически получают более высокий score, чем нормальные.

Главная проблема — FPR 0.54 и precision 0.48. Более половины срабатываний приходится на нормальные фреймы, которые модель посчитала аномальными. Это прямое следствие двух факторов:

1. **Синтетический фон неоднороден по интенсивности.** Нормальный трафик генерируется со случайными параметрами, некоторые нормальные потоки случайно попадают в область высокого score.
2. **Порог не откалиброван под операционные требования.** При `contamination=0.1` и `threshold_quantile=0.99` система изначально помечает около 10% потоков как аномальные — при реальной доле аномалий значительно ниже это ведёт к завышенному FPR.

## 3.10 Проверка устойчивости и нагрузочное тестирование

**Устойчивость к вариациям генерации.** Синтетические данные генерируются с элементами случайности (временны́е сдвиги, случайный выбор адресов и портов). Для оценки вариативности результатов необходимо запускать несколько прогонов с разными seed и сравнивать разброс метрик. В базовом прогоне данной работы использован один вариант датасета. Это является ограничением текущего эксперимента: для академически строгого вывода необходима оценка дисперсии по 3–5 независимым прогонам.

**Устойчивость порогов.** Пороги `t_low=0.611` и `t_high=0.858` вычислены по квантилям 90-го и 99-го процентилей на обучающей выборке нормального трафика. При изменении фона (другая нагрузка, другой сегмент сети) эти значения могут смещаться. В прототипе смещение диагностируется по метрике `ndr_suspicious_rate`: если доля non-regular фреймов резко растёт при стабильной конфигурации, это сигнал необходимости переобучения с обновлёнными порогами.

**Нагрузочное тестирование.** В рамках ВКР нагрузочное тестирование проводилось на синтетических данных объёмом до 3 000 фреймов. Время полного прогона (preprocessing + train + validate) укладывается в единицы секунд на стандартном ноутбуке. Это подтверждает применимость прототипа для офлайн-анализа; для онлайн-режима необходима дополнительная проверка на реальном трафике с измерением latency per frame.

Таблица 3.12 — Характеристики производительности прототипа

| Этап | Объём данных | Время выполнения | Ограничение |
|---|---|---|---|
| Генерация синтетики | 600–3000 записей | < 1 с | — |
| Предобработка | 300 фреймов | < 2 с | Линейно по числу фреймов |

**Поведение при смене фона.** Для проверки устойчивости к дрейфу была применена модель, обученная на данных одного часа, к данным другого часа. Это поведение является нормой для безучительских моделей и должно управляться через мониторинг `ndr_suspicious_rate`.


## 3.11 Анализ ложных срабатываний и пропусков

**Источники ложных срабатываний (FP).** Значение FPR=0.54 означает, что примерно каждый второй нормальный фрейм неверно помечается как подозрительный. Анализ показывает три основных источника:

1. **Граничные потоки синтетической нормы.** Нормальный трафик генерируется со случайными параметрами, часть потоков случайно попадает в верхнюю часть распределения по `pps` или `bps`. При `contamination=0.1` модель ожидает 10% аномалий и соответственно формирует пороги. Если реальная доля аномалий ниже, порог оказывается занижен.

2. **Одиночные пакеты с завышенной интенсивностью.** Для потоков из 1–2 пакетов `pps` и `bps` вычисляются через деление на `EPS_DUR=0.001 сек`, что даёт значения порядка 1000–10000. Такие наблюдения повышают score даже при отсутствии реальной аномалии.

3. **Фиксированный порог без калибровки.** Пороги `t_low` и `t_high` вычислены по квантилям обучающей выборки и не адаптированы к целевому FPR. Для снижения FPR до приемлемых 5–10% необходимо либо поднять `threshold_quantile` до 0.995–0.999, либо использовать адаптивный порог с валидацией на отложенной выборке.

**Направления снижения FPR:**

Таблица 3.13 — Меры по снижению ложных срабатываний

| Мера | Ожидаемый эффект | Стоимость |
|---|---|---|
| Поднять `threshold_quantile` до 0.995–0.999 | Снижение FPR при умеренной потере recall | Минимальная (изменение конфигурации) |
| Добавить признак числа уникальных dst_port за окно | Улучшение разделения scan от нормы | Требует изменения `extract_frame_features` |
| Фильтрация потоков из 1–2 пакетов | Устранение артефактов `pps/bps` | Минимальная (правило предфильтрации) |
| Включить AutoEncoder как отдельный классификатор и сравнить FPR | Потенциальное нелинейное разделение | Требует дополнительных прогонов |
| Переобучение на более широком нормальном окне | Расширение базовой линии | Нужен больший объём нормального трафика |

**Анализ пропусков (FN).** Recall 0.97 означает, что пропущено около 3% аномальных фреймов. Это фреймы, которые попали в окно burst-интервала, но статистически оказались близки к нормальному распределению. Такое возможно на границах аномального интервала, где в фрейм попало мало аномальных пакетов. При более строгой разметке (например, фрейм аномальный только если доля аномальных пакетов превышает 50%) recall снизился бы, но пропуски стали бы менее критичными по природе.

**Ожидаемое поведение на scan-сценарии.** Для port scan сигнал иного характера: не рост интенсивности, а рост числа уникальных целевых портов и доля SYN-пакетов. При текущем наборе признаков `syn_flag_ratio` включён, однако признак числа уникальных портов назначения отсутствует. Это означает, что recall на scan-сценарии будет ниже, чем на burst. Добавление признака `dst_port_entropy` или `unique_dst_ports_per_window` является приоритетным расширением для следующей итерации.

**Роль углубленной проверки в снижении FP.** Правило `rule:tcp_syn_low_ports_short_flow` частично компенсирует слабость ML-слоя на scan: оно срабатывает по структурному признаку (короткая SYN-попытка на привилегированный порт), а не по скоровой оценке. Однако для статуса `malicious` требуется совпадение и правила, и высокого score, что дополнительно снижает число ложных вердиктов высшей степени тяжести.

## 3.12 Выводы по главе 3

В третьей главе описана полная технологическая реализация прототипа и проведена его экспериментальная оценка. Полученные результаты позволяют сделать несколько содержательных выводов.

**Архитектура реализуема и воспроизводима.** Конвейер от захвата пакетов через Scapy до сохранения события в JSONL работает как единая цепочка с явными артефактами на каждом этапе. Изменение признаков или порогов не требует повторного захвата трафика: достаточно запустить pipeline начиная с этапа preprocessing на тех же Parquet-файлах. Все управляемые параметры вынесены в `pipeline_config.yaml`, что делает сравнение вариантов инженерно корректным.

**Детекция burst работает с высоким recall.** ROC-AUC 0.955 и recall 0.97 подтверждают, что безучительский ансамбль на метаданных L3-L4 уверенно ранжирует всплески интенсивности. Это достигается без доступа к содержимому пакетов, что делает подход применимым к шифрованному трафику.

**FPR требует калибровки.** Значение FPR=0.54 неприемлемо для промышленной эксплуатации, однако оно объяснимо и управляемо. Три конкретных меры — поднятие `threshold_quantile`, фильтрация артефактов одиночных потоков и добавление признаков структуры соединений — прогнозируемо снизят FPR до 10–15% при сохранении recall выше 0.9.

**Двухуровневый подход (ML + правила) правильный.** Разделение на `suspicious` (только ML-сигнал) и `malicious` (ML + правиловое подтверждение) предотвращает автоматические блокировки по ложным срабатываниям. В реальном контуре реагирования именно вердикт `malicious` с явным `evidence` является надёжным основанием для действия, а не сырой score.

**Применимость в реальных задачах.** Прототип занимает корректную нишу: он преобразует поток сетевой телеметрии в структурированное событие с оценкой риска, объяснимыми признаками и машиночитаемым контекстом. Это именно тот формат, который ожидают SIEM-системы и SOC-процессы на входе. Прототип не заменяет сигнатурные IDS или NGFW, но обеспечивает дополнительный поведенческий слой, устойчивый к шифрованию и новым видам атак.
